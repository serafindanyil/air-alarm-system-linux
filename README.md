# Автономна система моніторингу повітряних тривог

## Загальна ідея проєкту

Основною метою цього проєкту було створення автономної Embedded Linux системи. Ідея проста: система завантажується, автоматично піднімає набір Linux-сервісів, стукає в HTTP API за інформацією про повітряні тривоги і виводить поточний стан на зовнішній світлодіод через GPIO Raspberry Pi 4.

Оскільки це тестове завдання, я вирішив не просто написати прикладну логіку і забути, а пройти повноцінний цикл створення Embedded-системи: від розробки софту на C++ до генерації власного образу ОС через Buildroot.

---

## Хід розробки та організація простору

Я вирішив тримати все в одному монорепозиторії. Тут лежить:

* `BR2_EXTERNAL` пакет Buildroot;
* Вихідний код сервісів;
* Root filesystem overlay;
* Конфігураційні файли Buildroot;
* Docker-оточення для збірки.

Працював я переважно через SSH безпосередньо на Raspberry Pi 4, де стояв офіційний дистрибутив. Сконфігурував усе для C++, написав `CMakeLists`, підключив дебагер і розділив workspace на дві папки: `air-alarm-service` та `internet-monitor-service`, щоб вони не заважали одна одній.

Щоб не зациклюватися на багах, коли намагаєшся написати все й одразу, я розділив роботу на дрібні кроки. Кожна таска — окрема гілка і один PR. Спочатку налаштував середовище, потім написав базовий `main` з обробкою SIGINT сигналів (щоб сервіси вмирали красиво), далі зібрав модулі конфігурації, налаштував State Machine і прикрутив логування. Проста модульна архітектура, де кожен відповідає за своє.

## Збірка Image для Raspberry Pi 4

Я загорнув процес збірки в Docker. У корені репозиторію є папка `docker` з `docker-compose.yml` та `Dockerfile`. Контейнер сам копіює root, ставить залежності в Ubuntu, маунтить `BR2_EXTERNAL` і білдить проєкт через `make`.

Для запуску збірки достатньо виконати:

```bash
docker compose up --build

```

Якщо білд пройшов успішно, в кінці випаде такий лог:

```text
buildroot-1  | INFO: hdimage(sdcard.img): adding primary partition 'boot' (in MBR) from 'boot.vfat' ...
buildroot-1  | INFO: hdimage(sdcard.img): adding primary partition 'rootfs' (in MBR) from 'rootfs.ext4' ...
buildroot-1  | INFO: hdimage(sdcard.img): adding primary partition '[MBR]' ...
buildroot-1  | INFO: hdimage(sdcard.img): writing MBR
buildroot-1  | INFO: cmd: "rm -rf "/workspace/buildroot-output/build/genimage.tmp/"" (stderr):
buildroot-1 exited with code 0

```

Готовий образ буде чекати у `output/images/sdcard.img`.

## Інструкція: Як записати Image на SD-карту

Зібраний `sdcard.img` треба перенести на залізо. Вставляємо SD-карту в кардрідер і робимо наступне.
  
**Якщо ви адекватна людина на Linux:**

```bash
lsblk
sudo umount /dev/sdX* sudo dd if=output/images/sdcard.img of=/dev/sdX bs=4M status=progress conv=fsync
sync

```

*(де `/dev/sdX` — назва вашого диску з SD-картою)*

**Якщо ви, як і я, працюєте на MacOS:**

```bash
diskutil list
diskutil unmountDisk /dev/diskX
sudo dd if=output/images/sdcard.img of=/dev/rdiskX bs=4m status=progress
diskutil eject /dev/diskX

```

## Архітектура сервісів та логіка програми

Проєкт розбито на два окремих `systemd` сервіси.

### 1. Internet Monitor Service

Робить одну річ: перевіряє, чи є інтернет. Раз на 3 секунди пінгіує `8.8.8.8` і кешує результат. Він нічого не знає про повітряні тривоги і може бути перевикористаний у будь-якому іншому проєкті.

### 2. Air Alarm Service

Тут живе вся бізнес-логіка. Сервіс автоматично стартує після ініціалізації мережі. Робить запити на API (дефолтно кожні 10 секунд), розбирає JSON і смикає методи перевірки тривоги для поточної та сусідніх областей. Керування логікою реалізовано через State Machine, яка змінює стани і передає команди на зовнішній LED.

### Міжпроцесна взаємодія

Сервіси спілкуються між собою через **D-Bus**. Мені довелося окремо писати policy, щоб дозволити ці зв'язки. Чому я обрав D-Bus, а не простий спільний файл чи Unix Socket? Відверто кажучи — банальна цікавість. Хотілося реалізувати отримання статусу через повноцінний метод. Зараз я просто виводжу в канал `true` або `false` — це працює, але архітектурно для такої задачі спільний файл був би простішим і правильнішим рішенням.

### Алгоритм суміжних областей

Щоб коректно і швидко визначати сусідів, я реалізував список суміжності. Це дало складність `O(1)`. Максимально продуктивно і без зайвих ускладнень. Якби області України раптом стали динамічними, було б раціональніше підняти граф, виставити початковий pivot і при кожній зміні перераховувати відстані через DFS, записуючи сусідів у вектор чи `unordered_map`. Але оскільки області статичні, `O(1)` — наш найкращий друг.

## Конфігурація

Конфігураційний файл — це звичайний JSON. Він містить налаштування API, регіону та світлодіода.

Щоб відкрити конфіг, введіть:

```bash
air-alarm-config

```

### Структура конфігураційного файлу

| Поле | Тип | Обов'язкове | Опис | Приклад |
| --- | --- | --- | --- | --- |
| **api.url** | string | Так | URL API повітряних тривог. | `"https://ubilling.net.ua/aerialalerts/"` |
| **api.poll_interval_seconds** | integer | Так | Інтервал опитування (понад 3 сек). | `10` |
| **region.name** | string | Так | Назва області для моніторингу. | `"Донецька область"` |
| **led.gpio_pin** | integer | Так | Номер GPIO-піну світлодіода. | `17` |
| **led.gpio_chip_path** | string | Так | Шлях до GPIO-чипа в Linux. | `"/dev/gpiochip0"` |
| **led.active_alarm_blink_hz** | number | Так | Частота блимання (Гц) при прямій тривозі. | `2` |
| **led.neighbor_alarm_blink_hz** | number | Так | Частота блимання (Гц) при тривозі у сусідів. | `0.5` |

**Приклад `config.json`:**

```json
{
  "api": {
    "url": "https://ubilling.net.ua/aerialalerts/",
    "poll_interval_seconds": 10
  },
  "region": {
    "name": "Донецька область"
  },
  "led": {
    "gpio_pin": 17,
    "gpio_chip_path": "/dev/gpiochip0",
    "active_alarm_blink_hz": 2,
    "neighbor_alarm_blink_hz": 0.5
  }
}

```

Зв'язки між областями захардкоджені у файлі. Щоб подивитись список доступних областей, є команда:

```bash
air-alarm-list

```

## Корисні команди

```bash
air-alarm-config                             # Відкрити налаштування
air-alarm-list                               # Показати всі доступні області
systemctl status air-alarm-service           # Перевірити статус основного сервісу
systemctl status internet-monitor-service    # Перевірити статус монітору мережі
systemctl {start|stop|restart} {service}     # Керування процесами
journalctl -u air-alarm-service -f           # Читати логи в реальному часі

```

Якщо використовуєте звичаний дистрибутив від Raspberry Pi 4, дуже корисна команда:

```bash
pinout # Щоб отримати розпіновку плати
```

## Що б я покращив у майбутньому

Немає межі досконалості, і код на C++ можна рефакторити вічно, але якщо виходити за поточний скоуп задач, я б зробив наступне:

1. **Тротлінг та кешування Інтернету:** Моніторинг мережі варто робити лише тоді, коли це необхідно (за запитом через метод), щоб не спамити пінг-запитами, а віддавати закешований результат.
2. **Індикація життєвого циклу:** Додав би окрему індикацію світлодіодом при ініціалізації сервісу та його зупинці, щоб мати візуальний фідбек про старт/стоп.
3. **Зміна стеку для бізнес-логіки:** Можливо, для самої логіки сервісу краще було б використати скриптову мову (наприклад, Python) або скомбінувати підходи. Це б значно пришвидшило розробку і дозволило б сфокусувати більше уваги саме на конфігурації Linux-системи, а не на боротьбі з пам'яттю.
4. **Масштабування індикації:** Додати підтримку кількох LED, звукову сигналізацію або вивід даних на OLED-дисплей.
